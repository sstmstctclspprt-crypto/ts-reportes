1. Arquitectura general del sistema

Vista de alto nivel: qué hace cada plataforma. Vercel solo publica la PWA (HTML/JS/CSS). Supabase concentra autenticación, base de datos, almacenamiento de archivos y funciones serverless (Edge Functions). Google autentica usuarios y guarda PDFs en Drive. Power Automate copia el PDF a OneDrive como respaldo.

Vercel

¿Qué es y qué NO hace?

Hosting del frontend Vue/Vite. Entrega la PWA por HTTPS. No procesa PDFs, no guarda datos ni ejecuta lógica de negocio.

vercel.json · frontend/dist

Supabase

Corazón del backend

Auth (sesiones JWT), tablas con RLS, buckets de Storage y Edge Functions en Deno. Toda la lógica de registros C-TPAT vive aquí.

backend/supabase/

Google

Identidad + archivo final

OAuth con scope drive.file. El PDF definitivo se sube a la cuenta Google del operador en carpetas TS REPORTES/PDFs.

Power Automate

Respaldo OneDrive

Tras generar el PDF, la Edge Function notifica un webhook. Power Automate descarga el PDF (URL firmada) y lo guarda en OneDrive según subcarpeta configurada.

3. Flujo de autenticación y control de acceso

No basta con iniciar sesión en Google: cada usuario pasa por control de acceso en Supabase (código de invitación o aprobación manual del administrador). Las políticas RLS bloquean datos hasta estar approved. 

sequenceDiagram
  autonumber
  participant U as Usuario (PWA)
  participant V as Vercel (solo entrega app)
  participant G as Google OAuth
  participant SA as Supabase Auth
  participant DB as Supabase DB (RPC)

  U->>V: Abre tacticalsupport.online
  V-->>U: Entrega PWA estática
  U->>SA: signInWithOAuth(google, scope drive.file)
  SA->>G: Redirect login
  G-->>SA: Callback + tokens
  SA-->>U: JWT en localStorage (PKCE)
  U->>DB: RPC sync_user_access_context()
  alt Email en app_admin_emails
    DB-->>U: status=approved, is_admin=true
  else Usuario nuevo
    DB-->>U: status=pending
    U->>U: Redirect /activacion
    U->>DB: RPC redeem_access_code(TS-XXXXXXXX)
    DB-->>U: status=approved
  else Admin manual
    Note over U,DB: AdminView → admin_set_user_access
    DB-->>U: approved / rejected
  end
  U->>DB: CRUD registros (RLS is_user_approved)

4. Flujo completo: registro → evidencias → PDF → Drive → OneDrive

Flujo principal de negocio. El usuario llena el formulario C-TPAT, sube fotos y firmas, se inserta en BD y se encola la generación del PDF. La Edge Function generate-ctpat-pdf arma el documento, lo sube a Google Drive y notifica Power Automate.

5. Flujo offline y cola de sincronización

Si no hay internet, el registro se guarda en IndexedDB (cola local). Al reconectar, syncStore.processQueue inserta en Supabase y dispara la Edge Function. El Service Worker de Vercel cachea el shell de la app para abrir sin red.

7. Retención de datos y purga automática

Los datos sensibles en BD se conservan ~7 días y luego se eliminan la fila completa. Los PDFs en Google Drive, OneDrive y Storage no se borran automáticamente por el cron de purga.

  B --> C[sanitizeRegistroSensitiveColumns

  C --> D[Firmas y image_urls siguen en BD

  D --> E{¿expires_at vencido?}
